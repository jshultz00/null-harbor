# config/fw-core/ — Core Internal Firewall

This directory contains `nftables.conf` for the `fw-core` container. fw-core is the internal routing/segmentation firewall. It sits at the intersection of all internal segments and routes traffic between DMZ, server, user, DB, and SIEM segments.

---

## Interface Layout

| Interface | Segment | Subnet | IP |
|-----------|---------|--------|-----|
| `eth0` | management (OOB — not visible to participants) | 10.0.0.0/24 | 10.0.0.11 |
| `eth1` | dmz | 10.10.10.0/24 | 10.10.10.2 |
| `eth2` | server | 10.20.20.0/24 | 10.20.20.1 |
| `eth3` | users | 10.30.30.0/24 | 10.30.30.1 |
| `eth4` | db | 10.40.40.0/24 | 10.40.40.1 |
| `eth5` | siem | 10.50.50.0/24 | 10.50.50.1 |

IPv4 forwarding enabled via sysctl.

---

## Routing Intent

fw-core is deliberately **permissive** — the goal is to generate realistic traffic logs for defenders to analyze, not to prevent lateral movement. All inter-segment traffic is logged. The routing table:

| Source Segment | Destination Segment | Policy | Notes |
|----------------|--------------------|---------|----|
| dmz | server | accept + log | Web servers accessing AD for auth |
| dmz | db | accept + log | Web-to-DB traffic |
| users | server | accept + log | Workstation-to-DC, workstation-to-exchange |
| users | dmz | accept + log | User browsing internal web |
| server | db | accept + log | App servers querying MSSQL |
| * | siem | accept + log | Syslog and Wazuh traffic |
| management | * | accept, **no log** | Silently accepted before log rule — never visible in SIEM |
| * | management | accept, **no log** | Silently accepted before log rule — never visible in SIEM |

> **Invariant:** Management traffic (eth0 / 10.0.0.0/24) must never appear in `FW-CORE-FWD:` log lines. This is enforced by accepting it before the `log` statement in the forward chain. Defenders must not be able to discover the OOB management plane from firewall logs.

## nftables Ruleset Structure

```nftables
table inet filter {

    chain input {
        type filter hook input priority 0; policy drop;

        ct state established,related accept

        # Management interface — accepted before any logging
        iifname "eth0" accept

        # ICMP
        ip protocol icmp accept
        ip6 nexthdr icmpv6 accept
    }

    chain forward {
        type filter hook forward priority 0; policy accept;

        # Drop invalid state
        ct state invalid drop

        # Management traffic — accepted silently before any logging
        # 10.0.0.0/24 must never appear in FW-CORE-FWD log lines
        iifname "eth0" accept
        ip daddr 10.0.0.0/24 accept

        # Log all inter-segment traffic (defenders analyze these)
        log prefix "FW-CORE-FWD: " flags all
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

### Design Note: Why No Blocking Rules?

For incident response training, defenders must respond to alerts generated by real attack traffic traversing the network. If fw-core silently blocks lateral movement attempts, there are no logs, no Wazuh alerts, and nothing for blue team to find. All blocking in this range is a blue team action (they reconfigure fw-core during the exercise), not a pre-configured range restriction.

## Logging

All forwarded traffic is logged with prefix `FW-CORE-FWD:`. Logs flow:

```
fw-core container stdout
  → Docker log driver
    → rsyslog container (10.50.50.8:514)
      → Wazuh manager filebeat
        → Wazuh indexer
          → Wazuh dashboard
```

Key log patterns Wazuh rules should detect:
- `FW-CORE-FWD:` with `daddr 10.20.20.100` on non-standard ports (credential stuffing toward DC)
- High-frequency new connections from single source (lateral movement scanning)
- SMB (`dport 445`) from user segment to server segment (ransomware precursor)

Management traffic (10.0.0.0/24) is silently accepted before the log rule and must never appear in these logs.
