# config/wireguard/ — WireGuard VPN Server

This directory is bind-mounted read-write into the `linuxserver/wireguard` container. The container writes generated peer configs back here so they persist across restarts and can be distributed to participants.

---

## Directory Contents

| Path | Description |
|------|-------------|
| `wg0.conf` | WireGuard server interface configuration |
| `peers/` | Generated peer configs (one `.conf` per participant, created by `make vpn-config`) |
| `peers/registry.txt` | Simple log of peer name → public key → assigned IP |

All files under `peers/` are gitignored — peer configs contain private keys and should never be committed.

---

## wg0.conf — Server Interface

```ini
[Interface]
Address = 10.99.0.1/24
ListenPort = 51820
PrivateKey = <server private key — generated at first start, stored in ./data/wireguard/>

# Route all range segments to VPN peers
PostUp   = iptables -A FORWARD -i wg0 -j ACCEPT; \
           iptables -A FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -A POSTROUTING -s 10.99.0.0/24 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; \
           iptables -D FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -D POSTROUTING -s 10.99.0.0/24 -j MASQUERADE
```

The server private key is auto-generated by `linuxserver/wireguard` on first start and stored in the container's volume. The public key is derived from it and printed to container logs on startup.

## Participant Access Model

When a participant connects via WireGuard, they receive:
- A `10.99.0.x` IP on the VPN segment
- Routes to exercise segments: `10.10.10.0/24, 10.20.20.0/24, 10.30.30.0/24, 10.40.40.0/24, 10.50.50.0/24, 10.99.0.0/24, 9.53.99.0/24`
- DNS resolution via the WireGuard container at `10.99.0.1` (which proxies to CoreDNS internally)

From their VPN IP, participants can:
- RDP to any Windows machine by segment IP (`mstsc /v:10.30.30.20`)
- SSH to any Linux machine by segment IP (`ssh devuser@10.30.30.10`)
- Access Wazuh dashboard at `https://10.50.50.7`
- Browse the fake internet via the scenario container's Caddy HTTPS server

> **Note:** The management network (10.0.0.0/24) is **not** included in participant routes. Participants reach Wazuh at its SIEM-segment IP (`10.50.50.7`) and send DNS queries to `10.99.0.1` (the WireGuard container), which proxies to CoreDNS internally. No 10.0.0.x address is ever visible to participants.

## Peer Config Generation — `make vpn-config PEER=alice`

The Makefile target calls into the `linuxserver/wireguard` container which:

1. Generates a keypair for `alice` (`wg genkey | tee peer-priv | wg pubkey > peer-pub`)
2. Assigns the next available IP in `10.99.0.0/24` (starting at `.2`, skipping `.1` = server)
3. Writes `config/wireguard/peers/alice.conf`:

```ini
[Interface]
PrivateKey = <alice's private key>
Address = 10.99.0.2/32
DNS = 10.99.0.1

[Peer]
PublicKey = <server's public key>
Endpoint = <host LAN IP>:51820
AllowedIPs = 10.10.10.0/24, 10.20.20.0/24, 10.30.30.0/24, 10.40.40.0/24, 10.50.50.0/24, 10.99.0.0/24, 9.53.99.0/24
PersistentKeepalive = 25
```

4. Adds alice's public key to `wg0.conf` as a new `[Peer]` block
5. Hot-reloads the WireGuard interface: `wg addconf wg0 <(wg-quick strip wg0)` or `wg syncconf`
6. Prints the config as a QR code via `qrencode` (for mobile import) and saves the file

## Peer Lifecycle

- Peer configs **persist across `make down`** because the volume mount is outside Docker's managed volumes
- Peer configs are **wiped by `make reset`** only if `data/wireguard/` is included in the reset; by default participant configs survive range resets so participants don't re-enroll between exercises
- To revoke a participant: remove their `[Peer]` block from `wg0.conf` and run `wg syncconf wg0 <(wg-quick strip wg0)` on the container

## Range DNS

Peer configs set `DNS = 10.99.0.1` — the WireGuard server's own VPN-facing IP. The WireGuard container runs a lightweight DNS forwarder (dnsmasq or similar) on `10.99.0.1:53` that proxies queries to CoreDNS on the scenario container (`10.0.0.1:53`) over the management network internally.

This keeps the management network completely hidden: participants send DNS queries to `10.99.0.1` (VPN segment, already in their routes) and never directly contact anything on `10.0.0.0/24`.

CoreDNS resolves all range hosts to their *segment* IPs, not management IPs:
- `dc01.secure.net` → `10.20.20.100`
- `exchange.secure.net` → `10.20.20.10`
- `wazuh.secure.net` → `10.50.50.7`
- `web-lin` → `10.10.10.10`
- etc.

DNS responses must never contain `10.0.0.x` addresses.

CoreDNS is configured in the scenario container's Dockerfile. Zone file is generated at container start from the machine IP table.
